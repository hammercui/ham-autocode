## Feature 2: OpenCode 子 Agent

### 目标
将 OpenCode（本机已安装，配置 GLM-5.1 模型）作为第五个执行目标，利用免费/低成本国产模型执行低复杂度任务，节省 Claude API 费用。

### 当前 Agent 矩阵

| Agent | 模型 | 成本 | 适合 |
|-------|------|------|------|
| Claude Code | claude-opus-4-6 | 高 | 复杂架构、多文件联动 |
| Codex | GPT-5 | 中 | 需求清晰的独立任务 |
| Claude App | claude-sonnet | 中 | 轻量（文档/配置/热修复） |
| Agent Teams | claude-sonnet×N | 高 | 并行多任务 |
| **OpenCode** (NEW) | GLM-5.1（智谱） | **免费** | 简单修复、格式化、重命名、文档更新 |

### 实现方案

**路由规则扩展：**
```
RoutingTarget: 'claude-code' | 'codex' | 'claude-app' | 'agent-teams' | 'opencode'

路由规则（新增）：
  complexityScore <= 20 AND files.length <= 3 → opencode（最简单的任务）
  specScore >= 80 AND isolationScore >= 70    → codex
  task type in [doc, config, hotfix]          → claude-app
  complexityScore >= 70                       → claude-code
  else                                        → claude-code (默认)
```

**Executor Adapter：**
```typescript
// core/executor/opencode.ts
export class OpenCodeAdapter extends BaseAdapter {

  }
}
```

**Agent 定义：**
```markdown

tools: [Read, Write, Edit, Bash, Glob, Grep]
```

## Feature 3: Token 节省（智能上下文）

### 目标
通过三层策略减少 token 消耗，让 harness 在长会话中保持高效。

### 三层方案

#### Layer 1: 文件摘要缓存（零依赖，立即可用）

```typescript
// core/context/summary-cache.ts

interface FileSummary {
  path: string;
  hash: string;          // 文件内容 hash
  tokens: number;        // 原始 token 数
  summary: string;       // 压缩摘要（提取 exports + 类/函数签名）
  summaryTokens: number; // 摘要 token 数
  savedPercent: number;  // 节省比例
}

/**
 * 对大文件（>200 tokens）自动提取摘要：
 * - TypeScript/JavaScript: 提取 export/class/function/interface 签名
 * - Markdown: 提取标题 + 第一段
 * - JSON: 提取顶层 key 结构
 *
 * 缓存到 .ham-autocode/context/summaries.json
 * 文件未变化时复用缓存（基于 hash 比较）
 */
export function summarizeFile(filePath: string): FileSummary
export function getSummaryOrFull(filePath: string, tokenBudget: number): string
```

**集成点：** `context prepare` 使用摘要而非全文，节省 40-60% token。

#### Layer 2: 增量上下文（只加载变化的部分）

```typescript
// core/context/incremental.ts

/**
 * 跟踪上下文中已加载的文件和版本。
 * 后续 prepare 只加载变化的文件差异，而非全文重载。
 */
interface ContextSnapshot {
  files: Map<string, { hash: string; loadedAt: string }>;
}

export function prepareIncremental(
  task: TaskState,
  previousSnapshot: ContextSnapshot
): { newFiles: string[]; changedFiles: string[]; unchanged: string[] }
```

#### Layer 3: 语义检索（MCP 集成，可选）

利用已有 MCP 生态做语义搜索，按需加载最相关的代码片段。

```
方案 A: CodeGrok MCP（向量检索）
  - 索引项目代码 → 向量数据库
  - context prepare 时按任务语义检索最相关片段
  - 比全文加载节省 80%+

方案 B: 零依赖 TF-IDF
  - core/context/tfidf.ts — 简单的 TF-IDF 实现
  - 对项目文件建索引
  - 按任务描述检索 top-K 相关文件
  - 不如向量检索精确，但零依赖
```

**推荐方案：** Layer 1（立即）+ Layer 2（中期）+ Layer 3 方案 B（简单 TF-IDF）。保持零运行时依赖原则。

## Feature 5: 项目进度管理增强

### 目标
引入软件工程中成熟的项目进度管理方法论，让 DAG 调度器从"简单的依赖图"升级为"专业的项目管理引擎"。

### 当前 DAG 的不足

| 能力 | 当前状态 | 改进方向 |
|------|---------|---------|
| WBS | ✅ parser 解析表格/markdown | 支持多级 WBS 编号（1.1.1 → parent/child 关系） |
| 依赖图 | ✅ blockedBy + topoSort | 关键路径分析（CPM） |
| 进度追踪 | ✅ dag status（百分比） | 甘特图 + 里程碑追踪 |
| 资源分配 | ⚠️ 路由到 agent | 资源负载均衡（agent 并行度管理） |
| 风险管理 | ⚠️ recovery 策略 | 风险登记 + 概率影响矩阵 |
| 估算 | ❌ 无 | 基于历史的工期估算（PERT） |

### 方法论来源

| 方法 | 来源 | 应用 |
|------|------|------|
| **CPM（关键路径法）** | 杜邦/Remington Rand 1950s | 识别最长路径，优化并行度 |
| **PERT（计划评审技术）** | 美国海军 1958 | 三点估算（乐观/最可能/悲观） |
| **EVM（挣值管理）** | DoD 1960s | SPI/CPI 进度和成本绩效指数 |
| **甘特图** | Henry Gantt 1910s | 可视化时间线 |
| **WBS 分解** | PMI PMBOK | 多级任务分解 |

### 实现方案

#### P1: 关键路径分析（CPM）

```typescript
// core/dag/critical-path.ts

interface CPMResult {
  criticalPath: string[];        // 关键路径上的 task IDs
  criticalPathDuration: number;  // 关键路径总工期（估算）
  slack: Map<string, number>;    // 每个任务的浮动时间
  parallelism: number;           // 最大并行度
  bottlenecks: string[];         // 瓶颈任务（slack=0 且多个后继）
}

/**
 * 基于 DAG 依赖图和工期估算，计算关键路径。
 * 工期来源：learning insights 的历史平均 或 complexityScore 推断
 */
export function analyzeCriticalPath(tasks: TaskState[]): CPMResult
```

**CLI：** `dag critical-path`

#### P2: PERT 三点估算

```typescript
// core/dag/estimation.ts

interface PERTEstimate {
  taskId: string;
  optimistic: number;    // 乐观（分钟）
  mostLikely: number;    // 最可能
  pessimistic: number;   // 悲观
  expected: number;      // (O + 4M + P) / 6
  stdDev: number;        // (P - O) / 6
}

/**
 * 基于历史数据和 complexityScore 生成三点估算。
 * 如果有历史同类任务数据（learning patterns），用实际值。
 * 否则用启发式：
 *   乐观 = complexityScore * 0.5 分钟
 *   最可能 = complexityScore * 1.0 分钟
 *   悲观 = complexityScore * 2.5 分钟
 */
export function estimatePERT(tasks: TaskState[], projectDir: string): PERTEstimate[]
```

**CLI：** `dag estimate`

#### P3: EVM 挣值分析

```typescript
// core/dag/earned-value.ts

interface EVMMetrics {
  PV: number;   // 计划价值（应完成的任务数）
  EV: number;   // 挣值（已完成的任务数）
  AC: number;   // 实际成本（已消耗的 token/时间）
  SPI: number;  // 进度绩效指数 = EV/PV（>1 提前，<1 延迟）
  CPI: number;  // 成本绩效指数 = EV/AC（>1 节省，<1 超支）
  EAC: number;  // 完工预估
  VAC: number;  // 完工偏差
  status: 'ahead' | 'on-track' | 'behind' | 'critical';
}

/**
 * 基于 pipeline 启动时间、当前进度、历史速率计算 EVM 指标。
 */
export function calculateEVM(projectDir: string): EVMMetrics
```

**CLI：** `dag evm`
