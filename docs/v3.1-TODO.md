## Feature 1: 竞品分析能力

### 目标
给 ham-autocode 增加对目标项目所在领域的竞品分析能力。在 `/ham-autocode:auto` 的 Phase 1（立项）中，自动搜索和分析竞品，辅助决策。

### 实现方案

**新增 Skill：** `/ham-autocode:research`

```
研究目标项目的竞品和技术生态：
1. 用 /research (gstack tavily) 搜索竞品
2. 用 agent-browser 打开竞品网站截图+分析
3. 用 /crawl 抓取竞品文档
4. 生成结构化竞品分析报告
```

**可用工具链（已安装）：**

| 工具 | 来源 | 能力 |
|------|------|------|
| `/research` | gstack (tavily) | Web 搜索 + 多源综合 |
| `/search` | gstack (tavily) | 快速单次搜索 |
| `/crawl` | gstack (tavily) | 爬取网站为 markdown |
| `/extract` | gstack (tavily) | 提取指定 URL 内容 |
| `agent-browser` | vercel-labs | CDP 浏览器自动化，截图+交互 |
| `/browse` | gstack | 无头浏览器 QA 测试 |

**Core Engine 集成：**

## Feature 2: OpenCode 子 Agent

### 目标
将 OpenCode（本机已安装，配置 GLM-5.1 模型）作为第五个执行目标，利用免费/低成本国产模型执行低复杂度任务，节省 Claude API 费用。

### 当前 Agent 矩阵

| Agent | 模型 | 成本 | 适合 |
|-------|------|------|------|
| Claude Code | claude-opus-4-6 | 高 | 复杂架构、多文件联动 |
| Codex | GPT-5 | 中 | 需求清晰的独立任务 |
| Claude App | claude-sonnet | 中 | 轻量（文档/配置/热修复） |
| Agent Teams | claude-sonnet×N | 高 | 并行多任务 |
| **OpenCode** (NEW) | GLM-5.1（智谱） | **免费** | 简单修复、格式化、重命名、文档更新 |

### 实现方案

**路由规则扩展：**
```
RoutingTarget: 'claude-code' | 'codex' | 'claude-app' | 'agent-teams' | 'opencode'

路由规则（新增）：
  complexityScore <= 20 AND files.length <= 3 → opencode（最简单的任务）
  specScore >= 80 AND isolationScore >= 70    → codex
  task type in [doc, config, hotfix]          → claude-app
  complexityScore >= 70                       → claude-code
  else                                        → claude-code (默认)
```

**Executor Adapter：**
```typescript
// core/executor/opencode.ts
export class OpenCodeAdapter extends BaseAdapter {
  generateInstruction(task: TaskState): string {
    // 生成 opencode 可执行的指令
    // opencode -p "..." --model glm-5.1
    return `opencode -p "${task.spec.description}" --model glm-5.1`;
  }
}
```

**Agent 定义：**
```markdown
# agents/opencode.md
---
name: opencode
description: Low-cost coding agent for simple tasks (GLM-5.1)
model: glm-5.1
tools: [Read, Write, Edit, Bash, Glob, Grep]
---
```

## Feature 3: Token 节省（智能上下文）

### 目标
通过三层策略减少 token 消耗，让 harness 在长会话中保持高效。

### 三层方案

#### Layer 1: 文件摘要缓存（零依赖，立即可用）

```typescript
// core/context/summary-cache.ts

interface FileSummary {
  path: string;
  hash: string;          // 文件内容 hash
  tokens: number;        // 原始 token 数
  summary: string;       // 压缩摘要（提取 exports + 类/函数签名）
  summaryTokens: number; // 摘要 token 数
  savedPercent: number;  // 节省比例
}

/**
 * 对大文件（>200 tokens）自动提取摘要：
 * - TypeScript/JavaScript: 提取 export/class/function/interface 签名
 * - Markdown: 提取标题 + 第一段
 * - JSON: 提取顶层 key 结构
 *
 * 缓存到 .ham-autocode/context/summaries.json
 * 文件未变化时复用缓存（基于 hash 比较）
 */
export function summarizeFile(filePath: string): FileSummary
export function getSummaryOrFull(filePath: string, tokenBudget: number): string
```

**集成点：** `context prepare` 使用摘要而非全文，节省 40-60% token。

#### Layer 2: 增量上下文（只加载变化的部分）

```typescript
// core/context/incremental.ts

/**
 * 跟踪上下文中已加载的文件和版本。
 * 后续 prepare 只加载变化的文件差异，而非全文重载。
 */
interface ContextSnapshot {
  files: Map<string, { hash: string; loadedAt: string }>;
}

export function prepareIncremental(
  task: TaskState,
  previousSnapshot: ContextSnapshot
): { newFiles: string[]; changedFiles: string[]; unchanged: string[] }
```

#### Layer 3: 语义检索（MCP 集成，可选）

利用已有 MCP 生态做语义搜索，按需加载最相关的代码片段。

```
方案 A: CodeGrok MCP（向量检索）
  - 索引项目代码 → 向量数据库
  - context prepare 时按任务语义检索最相关片段
  - 比全文加载节省 80%+

方案 B: 零依赖 TF-IDF
  - core/context/tfidf.ts — 简单的 TF-IDF 实现
  - 对项目文件建索引
  - 按任务描述检索 top-K 相关文件
  - 不如向量检索精确，但零依赖
```

**推荐方案：** Layer 1（立即）+ Layer 2（中期）+ Layer 3 方案 B（简单 TF-IDF）。保持零运行时依赖原则。
