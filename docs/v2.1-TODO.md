## Phase 2: Gap 修复（P0 — 阻塞级）

- [ ] **G1: Executor Adapter 接入 CLI** (Gap A1)
  - 新增 CLI 命令: `execute prepare <task-id>`
  - 根据 task.routing.target 调用对应 adapter.generateInstruction()
  - 输出结构化执行指令供 skill 使用
  - 新增 CLI 命令: `execute parse-result <task-id> <success|failure> [error-type]`

- [ ] **G2: topoSort 接入 scheduler** (Gap A2)
  - scheduler.nextWave 启动时先调用 topoSort 做环检测
  - 如果发现环依赖，返回 cycles 列表而不是静默忽略
  - dag status 输出中包含环检测结果

## Phase 3: Gap 修复（P1 — 重要）

- [ ] **G3: 验证失败自动 rollback** (Gap B4)
  - validateTask 返回 action='block' 时检查 task.recovery.strategy
  - 如果 strategy='checkpoint' 且有 checkpointRef，自动调用 rollbackToCheckpoint
  - 如果 strategy='worktree'，自动调用 removeWorktree
  - 在 validation result 中记录 rollback 结果

- [ ] **G4: pipeline.current_task 自动更新** (Gap A5)
  - dag init 后将第一个 next-wave 任务设为 current_task
  - dag complete/fail 后清空 current_task 或设为下一个
  - pipeline status 输出中显示 current_task

- [ ] **G5: blocked 状态自动推断** (Gap A6)
  - dag status 时扫描所有 pending 任务
  - 如果任务的 blockedBy 中有未完成的任务，状态显示为 blocked
  - dag next-wave 只返回真正可执行的 pending 任务（blockedBy 全部 done/skipped）

- [ ] **G6: Recovery 策略自动选择** (Gap A7)
  - 新增 CLI: `recover auto <task-id>`
  - 根据 task.scores.complexityScore 自动选择策略
    - < 70 → checkpoint
    - >= 70 → worktree
  - 读取 config.recovery.highRiskThreshold 作为阈值

## Phase 4: Gap 修复（P2 — 改善）

- [ ] **G7: Execution Trace 日志** (Gap B1)
  - 每个 CLI dispatch 调用追加一行到 `.ham-autocode/logs/trace.jsonl`
  - 格式: `{ "time": ISO, "command": "dag complete task-001", "result": "ok", "duration_ms": 12 }`
  - trace 文件自动按天轮转（超过 1MB 时重命名为 trace-YYYY-MM-DD.jsonl）

- [ ] **G8: file-index.json 持久化** (Gap B2)
  - context prepare 后将 file index 写入 `.ham-autocode/context/file-index.json`
  - 下次 prepare 时先检查缓存（如果 mtime 未变则复用）
  - 新增 CLI: `context index` 单独构建并持久化索引

- [ ] **G9: DAG 依赖推断** (Gap A3)
  - parser 解析任务后，扫描所有任务对的文件列表
  - 如果 task B 的文件是 task A 的输出，推断 B blockedBy A
  - 推断结果标记为 `{ inferred: true }` 以区分手动定义
