# ham-autocode v2.3 Plan — OpenSpec 集成

> **主题：集成 OpenSpec 规范层，补齐 Harness 的规范能力（97% → 99%）**

## 问题分析

ham-autocode 当前的最大实际痛点（ham-video 集成测试发现）：

```
所有 33 个任务 specScore = 30 → 0 个路由到 Codex
原因：parser 只提取任务名，没有规范（interface/acceptance/completeness）
```

当前架构缺失**规范层**：

```
gstack (决策：做什么)      ← 已集成
  → ??? (规范：怎么定义)   ← 缺失！
    → Superpowers (执行：怎么做) ← 已集成
      → Core Engine (调度+恢复+验证) ← 已实现
```

OpenSpec 填补的正是这个空缺：**在决策（gstack）和执行（Superpowers）之间，插入规范驱动的任务定义层。**

## OpenSpec 核心概念映射

| OpenSpec 概念 | ham-autocode 对应 | 集成方式 |
|--------------|------------------|---------|
| `openspec/specs/` | 系统行为规范（source of truth） | 读取用于 context prepare |
| `openspec/changes/` | 变更提案（一个 change = 一组 artifacts） | 映射到 DAG task |
| `proposal.md` | task.spec.description | 自动填充 |
| `specs/*.md` (delta) | task.spec.interface + acceptance | 提取 GIVEN/WHEN/THEN |
| `design.md` | task.context.requiredFiles | 提取涉及文件 |
| `tasks.md` | task.files + blockedBy | 提取子任务和文件 |
| `/opsx:propose` | `dag init` 前的规范生成 | 新增 `spec enrich` 命令 |
| `/opsx:apply` | execute prepare → skill 执行 | 已有，传入规范 |
| `/opsx:verify` | validate gates | 已有 |
| `/opsx:archive` | dag complete + 合并 delta specs | 新增 `spec sync` 命令 |

## 集成架构

```
新增流程（★ 标记新增步骤）：

dag init (parse plan → tasks)
  ↓
★ spec enrich <task-id>     ← 调用 OpenSpec 生成规范 artifacts
  ↓
route batch                  ← specScore 提升 → 更多任务路由到 Codex
  ↓
execute prepare <task-id>    ← 现在包含完整规范
  ↓
[agent 执行]
  ↓
validate <task-id>
  ↓
★ spec sync <task-id>       ← 完成后合并 delta specs 到 source of truth
  ↓
dag complete <task-id>
```

## 实施计划

### Phase 1: Spec Enrichment Engine

**新增模块：** `core/spec/enricher.ts`

```typescript
interface SpecEnrichResult {
  taskId: string;
  description: string;
  interface: string;      // 从 OpenSpec specs/*.md 提取
  acceptance: string;     // 从 GIVEN/WHEN/THEN scenarios 提取
  completeness: number;   // 0-100 基于 artifacts 完整度评分
  files: string[];        // 从 design.md/tasks.md 提取
}
```

核心功能：
1. 检测项目是否有 `openspec/` 目录
2. 如果有：读取 `openspec/specs/` 和 `openspec/changes/` 提取规范
3. 如果没有：基于任务名和上下文生成建议规范（best-effort）
4. 更新 task.spec 字段（description, interface, acceptance, completeness）
5. 重新计算 specScore

**CLI：** `spec enrich <task-id>` 或 `spec enrich-all`

### Phase 2: Spec Completeness Scoring

**修改模块：** `core/routing/scorer.ts`

当前 specScore 计算：
```typescript
// 旧：只看 completeness 字段（parser 设为 0）
specScore = spec.completeness || 30
```

改为：
```typescript
// 新：基于 spec 各字段的填充度计算
specScore = calculateSpecScore(task.spec)

function calculateSpecScore(spec: TaskSpec): number {
  let score = 0;
  if (spec.description && spec.description.length > 20) score += 25;
  if (spec.interface && spec.interface.length > 0) score += 25;
  if (spec.acceptance && spec.acceptance.length > 0) score += 25;
  if (spec.completeness >= 80) score += 25;
  return Math.min(score, 100);
}
```

### Phase 3: OpenSpec Directory Integration

**新增模块：** `core/spec/reader.ts`

读取 OpenSpec 目录结构：
```typescript
interface OpenSpecProject {
  hasOpenSpec: boolean;
  specs: { domain: string; path: string; requirements: number }[];
  changes: { name: string; path: string; status: string; artifacts: string[] }[];
}

function readOpenSpecProject(projectDir: string): OpenSpecProject
function readSpecForDomain(projectDir: string, domain: string): SpecContent
function readChangeArtifacts(projectDir: string, changeName: string): ChangeArtifacts
```

### Phase 4: Spec Sync（完成后合并）

**新增模块：** `core/spec/sync.ts`

任务完成后将 delta specs 合并到 source of truth：
```typescript
function syncSpec(projectDir: string, taskId: string): SyncResult {
  // 1. 找到对应的 openspec/changes/<change>/
  // 2. 合并 delta specs 到 openspec/specs/
  // 3. 标记 change 为 archived
}
```

**CLI：** `spec sync <task-id>`

### Phase 5: CLI + Skill 集成

**新增 CLI 命令：**
```

```

## 依赖关系

```
Phase 1 (enricher) → Phase 2 (scorer) → Phase 5 (CLI)
Phase 3 (reader) → Phase 1 (enricher uses reader)
Phase 4 (sync) → Phase 5 (CLI)
Phase 6 独立
```

Phase 1+3 可并行，Phase 2 依赖 Phase 1，Phase 4 独立。

## 注意事项

1. **零运行时依赖原则不变** — OpenSpec 是外部工具，ham-autocode 只读取它的目录结构和文件
2. **OpenSpec 是可选的** — 如果项目没有 `openspec/` 目录，enricher 使用 best-effort 启发式
3. **不依赖 OpenSpec CLI** — 只读取文件，不调用 `openspec` 命令
4. **向后兼容** — 不改变已有 CLI 接口，只新增 `spec` 命令
