# ham-autocode v2.2 Roadmap

> **主题：生产级 Harness 能力（92% → 97%）**
>
> v2.2 在 v2.1（TypeScript + Gap 修复）基础上，补齐生产级 Harness 的关键能力。
> 原 v3 计划中的高价值功能提前到 v2.2。

## 目标

将 ham-autocode 从"可用的 Harness"升级为"可靠的 Harness"，对标 claude-code-harness (Chachamaru127) 和 Stripe Minions。

## Phase 1: Observability（可观测性）

### O1: Structured Execution Trace
- v2.1 已有基础 trace.jsonl
- 升级为结构化追踪：每个 task 的完整生命周期（created → routed → prepared → executing → validating → done/failed）
- 包含 token 消耗、耗时、文件变更列表
- 支持 `trace query --task <id>` 按任务过滤

### O2: DAG 可视化（Terminal）
- 新增 CLI: `dag visualize`
- 输出 ASCII DAG 图（任务节点 + 依赖边 + 状态着色）
- 显示当前 wave 和已完成/失败/待执行的分布

### O3: Session Report
- 新增 CLI: `session report`
- 汇总当前会话：执行了多少任务、成功/失败比、总 token 消耗、总耗时
- 支持对比两次会话的进度差异

## Phase 2: Auto-commit（自动提交）

### AC1: Validate-then-Commit 闭环
- 验证通过后自动 `git add` 变更文件 + `git commit`
- commit message 由 executor adapter 从 task.name 生成
- 可通过 harness.json 配置 `"autoCommit": true|false`（默认 false）
- 保持 v2.0 的"推荐而非强制"原则：默认关闭，显式启用

### AC2: Commit 粒度控制
- 每个 task 一个 commit（atomic commits）
- commit message 格式: `feat(task-001): [task.name]`
- 失败 rollback 时自动 `git reset` 对应 commit

## Phase 3: Agent Teams 集成

### AT1: Agent Teams 作为第四路由目标
- routing.target 枚举扩展: `"codex" | "claude-code" | "claude-app" | "agent-teams"`
- 路由规则: 当一个 wave 中有 3+ 独立任务，且每个任务 isolationScore >= 70 → agent-teams
- 新增 executor adapter: `agent-teams.ts`

### AT2: Progressive Batching
- 第一轮分配 2 个 teammate（验证稳定性）
- 如果成功，第二轮扩展到 4-5 个
- 如果失败，回退到单 agent 逐任务执行

### AT3: Teammate File Ownership
- 从 DAG 的 task.files 自动推断 teammate 的文件范围
- 检测文件冲突：同一文件不能分配给两个 teammate
- 冲突时自动将冲突任务合并到同一 teammate

## Phase 4: Guardrail Rules（守护规则）

### GR1: 声明式规则引擎
- 规则定义在 `rules/` 目录下，每个规则一个 .ts 文件
- 规则接口: `{ id: string, check(context): RuleResult }`
- CLI: `rules check` 运行所有规则，`rules list` 列出规则

### GR2: 核心规则集（R01-R08）
- R01: 单文件不超过 500 行（超出建议拆分）
- R02: 每次 commit 不超过 10 个文件（超出建议拆分任务）
- R03: 测试覆盖率不低于当前值（只增不减）
- R04: 不允许 TODO/FIXME 进入 done 状态的任务
- R05: import 路径必须存在（防止死引用）
- R06: 敏感文件保护（.env, credentials.* 不允许出现在 git add 中）
- R07: context 预算不超过 critical 阈值时不允许启动新任务
- R08: 失败任务不超过总任务数 20%（超出暂停 pipeline）

## Phase 5: 完整 Schema 校验

### SV1: 运行时 JSON Schema 验证
- 每次读取 pipeline.json / task-*.json 时用 schemas/ 下的 JSON Schema 校验
- 使用轻量 JSON Schema 验证器（如 ajv-lite 或手写 —— 保持零运行时依赖原则）
- 校验失败时 throw 明确错误而非静默忽略

### SV2: Schema Migration 完善
- 实现 v2 → v3 migration（如果 task schema 变化）
- 自动检测 schemaVersion 并在读取时升级

## v3.0 保留项（不在 v2.2 范围）

以下功能推迟到 v3.0：

| 功能 | 原因 |
|------|------|
| Web Dashboard | 需要前端框架，偏离 CLI-first 原则 |
| Evaluation System（自动评分 + 回归测试） | 需要真实项目数据积累 |
| YAML 配置 | 需要引入 YAML parser 依赖，与零依赖原则冲突 |
| Branch-based Recovery（第三层） | checkpoint + worktree 已覆盖 99% 场景 |
| session_id 持久化 | pipeline status 已足够支持 resume |
| Progressive Compaction（上下文压缩） | 依赖 Claude API 能力演进 |

v2.2  → Observability + Auto-commit + Agent Teams + Guardrails (97%)
