# Project State Detection

You are analyzing an existing project to determine its current development phase.
Scan systematically, report factually, and recommend the next action.

## Detection Protocol

### Step 1: Scan Project Structure

Read the project root and key directories:
- `ls` the project root
- Check for: CLAUDE.md, PROJECT.md, README.md, docs/, .planning/, src/, app/, tests/
- Check git status: `git log --oneline -20`, `git status`

### Step 2: Check Each Phase

#### Phase 1: Project Initiation
Look for:
- `/office-hours` output (design docs with "Generated by /office-hours")
- Product definition documents
- Competitive analysis
- CEO/founder review records
- Files: `design-*.md`, `*product-definition*`, `*competitive*`, archive/ with review output

#### Phase 2: Requirements & Milestones
Look for:
- PROJECT.md (GSD) or equivalent requirement documents
- Milestone plans with acceptance criteria
- WBS (Work Breakdown Structure)
- .planning/ directory (GSD state)
- Files: `*requirement*`, `*milestone*`, `*WBS*`, `*roadmap*`

#### Phase 3: Phase Planning
Look for:
- Detailed execution plans (PLAN.md or equivalent)
- Technical architecture documents
- Task breakdowns with dependencies
- Files: `PLAN.md`, `*architecture*`, `*design-doc*` (technical)

#### Phase 4: Development Execution
Look for:
- Source code in src/ or app/
- Git history with feat/fix commits
- Task completion status in WBS or todo files
- Test code
- Code review records
- Count: completed tasks vs pending tasks

#### Phase 5: Review & QA
Look for:
- Code review records, bug/issue lists
- QA test results, VERIFICATION.md
- Acceptance criteria completion status

#### Phase 6: Release
Look for:
- PRs or release records
- CHANGELOG.md, git tags
- Deployment records

### Step 3: Output Diagnosis

```
## Project State Diagnosis: [project-name]

### Overall: Phase [X] - [phase-name] ([percentage]%)

### Phase Status:
| Phase | Status | Completion | Evidence |
|-------|--------|------------|----------|
| Phase 1 Initiation | [Done/Partial/Not Started] | [%] | [key files] |
| Phase 2 Requirements | ... | ... | ... |
| Phase 3 Planning | ... | ... | ... |
| Phase 4 Development | ... | ... | ... |
| Phase 5 Review | ... | ... | ... |
| Phase 6 Release | ... | ... | ... |

### Current Blockers:
- [specific issues]

### Recommended Next Steps:
1. [action + skill command]
2. [Claude Code tasks vs Codex tasks]
```

### Step 4: Route to Next Action via Core Engine

Use the v2 core engine CLI to route tasks:

```bash
# Parse plan into DAG tasks
ham-cli dag init <plan-file> [milestone] [phase]

# Route all tasks to appropriate executors
ham-cli route batch

# Check current DAG status
ham-cli dag status

# Route individual tasks
ham-cli route <task-id>
```

Based on diagnosis:
- Which phases to SKIP (already done)
- Which phase to START from
- Which tasks → Claude Code (complex/architectural)
- Which tasks → Codex (clear requirements: file path + interface + acceptance criteria)
- Which `/skill` commands to run next

## Rules

- NEVER recommend re-executing completed phases
- Equivalent documents count (e.g., `docs/06-WBS.md` = GSD ROADMAP)
- Projects not initialized with GSD are still valid — match by content, not by tool
- Be specific about evidence — cite actual file names and git commits
