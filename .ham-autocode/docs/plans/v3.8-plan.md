# v3.8 Plan: 实战暴露的 Bug 修复 + 改进项

> 来源: v3.7 ham-video 实战 (23 任务, 3 轮 execute auto)

## Bug 修复 (P0)

### B1: Codex `< file` stdin 并行超时
**现象**: 3 个 codex 并行 spawn 全部 10min 超时，但文件实际已写入
**根因**: codex CLI 的 `< file` stdin 在并行场景下不可靠，进程写完文件后继续做验证步骤不退出
**影响**: codex 任务首次总是超时，依赖 fallback 或重试兜底
**方案**: 
- 方案 A: codex 改为参数传 prompt（不用 stdin）—— 但 prompt 超长时可能触发 shell 参数限制
- 方案 B: codex 串行执行（concurrency=1），opencode 并行
- 方案 C: 超时前先检查文件是否已创建，如果已创建直接视为成功，kill 进程
- **推荐 C**: 最小改动，兼顾并行和可靠性

### B2: agent-exec.jsonl 数据污染
**现象**: execute stats 显示 277 次执行，大量是 ENOENT 错误数据（调试产生的）
**方案**: 添加 `execute stats --reset` 清除历史数据，或 `--since 24h` 过滤时间范围

### B3: auto-status currentTasks agent 字段不更新
**现象**: currentTasks 一直显示 `agent: "pending", status: "queued"`，spawn 后没更新为实际 agent
**方案**: executeTask 开始时 updateProgress currentTasks

## 改进项 (P1)

### I1: 超时前文件检查（智能超时）
codex/opencode 经常写完文件后进程不退出（做后续验证）。改为：
```
每 30s 检查一次 task.files 是否已创建
如果文件已创建且非空 → 等待 30s grace period → kill 进程 → 视为成功
```
预期效果: task-022 不再超时（文件在 ~5min 创建，但进程要跑 10min）

### I2: Codex 恢复测试
当前 codex 因 stdin 问题被 cooldown，需要验证 `< file` 在非并行场景下是否可靠。
如果不可靠，改为 `codex exec --full-auto "$(cat file)"` 方式。

### I3: execute stats 时间窗口过滤
```bash
ham-cli execute stats --since 1h    # 最近 1 小时
ham-cli execute stats --since 24h   # 最近 24 小时
ham-cli execute stats --reset       # 清除历史
```

### I4: auto-runner 波次间间隔
当前 Wave N 完成后立即启动 Wave N+1。如果上一波有超时/失败，应等待 cooldown 恢复后再继续，避免浪费 agent 调用。

### I5: auto-status 增加 ETA
```json
{
  "eta": "~3 min remaining",
  "avgTaskDurationSec": 123
}
```
基于已完成任务的平均耗时，估算剩余时间。

### I6: bundle 中添加"不要创建测试文件"指令
opencode 经常额外创建测试文件（如 queue.test.ts），这些文件不在 task.files 中，可能导致 tsc 错误。
在 Surgical Changes 约束中添加: "Do not create test files unless explicitly requested."
