# v3.7 Plan: 全自动执行 + 质量保障 + 限额调度

## 问题
1. 执行层：每波任务需要 Claude Code 手动编排 ~60 min，占 session 70% 时间
2. 质量层：auto 不检查产出质量，错误代码会级联到下游
3. 限额层：agent 限额耗尽后无等待调度，24h 不间断做不到
4. 规划层：DAG task spec 质量参差不齐，影响 agent 产出

## 目标
一条命令跑完整个 milestone，自动执行 + 质量门禁 + 限额容灾。

```bash
# Claude Code 或 Claude App 中一条命令触发
ham-cli execute auto
```

## 设计

### 执行流程

```
execute auto
  │
  ├── 1. dag next-wave → tasks[]
  │     如果为空 → 输出 "所有任务已完成" → 退出
  │
  ├── 2. 对每个 task 并行:
  │     ├── route task → 确定 target
  │     ├── execute prepare --raw → 写入临时文件
  │     ├── spawn agent (stdin < 临时文件)
  │     ├── 等待完成 (timeout: 10 min)
  │     ├── 验证: 检查 task.files 是否存在
  │     ├── 成功 → dag complete + execute log
  │     └── 失败 → fallback 链尝试下一个 agent
  │
  ├── 3. 本波全部完成 → git commit (按波提交)
  │
  ├── 4. 输出波次汇总
  │
  └── 5. 回到步骤 1
```

### Fallback 链（3 级）

```
推荐 codex    → 失败 → opencode → 失败 → skip + 记录
推荐 opencode → 失败 → codex    → 失败 → skip + 记录
```

注: claude-code subagent 不作为 fallback（因为是同账号限额，且需要在 Claude Code session 内才能调用，execute auto 是独立进程）。

### 自动 Git Commit

每波完成后自动提交：

```
git add <本波所有 task 涉及的文件>
git commit -m "feat: auto-execute wave N — task-001, task-002, task-003

Agent: codex(2), opencode(1)
Duration: 450s total
Files: 3 created, 1 modified"
```

规则：
- 只 add 任务声明的文件（task.files），不 add 无关文件
- commit message 自动生成，包含 agent 统计
- 不 push（push 由人决定，或加 --push 参数）

### CLI 接口

```bash
# 基本用法：自动跑完所有剩余任务 + 每波 commit
ham-cli execute auto

# 指定 agent（不走路由）
ham-cli execute auto --agent codex
ham-cli execute auto --agent opencode

# 单任务超时（默认 600000ms = 10min）
ham-cli execute auto --timeout 300000

# 干跑模式：只输出计划不执行
ham-cli execute auto --dry-run

# 最大并行数（默认 = wave 大小，即全部并行）
ham-cli execute auto --concurrency 3

# 完成后自动 push
ham-cli execute auto --push
```

### 触发方式

| 环境 | 触发命令 |
|------|----------|
| Claude Code | `Bash: ham-cli execute auto` (run_in_background) |
| Claude App | 在终端中执行 `HAM_PROJECT_DIR=... ham-cli execute auto` |
| 定时任务 | cron / Windows Task Scheduler |
| 手动 | 直接在终端跑 |

关键：`execute auto` 是一个独立 Node.js 进程，不依赖 Claude Code session。
Claude Code/App 只负责启动它，之后可以去做别的事。

### 失败处理

| 场景 | 处理 |
|------|------|
| agent 超时 (>timeout) | kill → fallback |
| agent 退出码非 0 | log error → fallback |
| fallback 也失败 | skip + 记录原因 |
| 文件未创建但进程成功 | warning + dag complete（agent 可能重命名） |
| 一波中部分失败 | 成功的照常 commit，失败的 skip |
| 全部 skip | 输出汇总 → 退出码 1 |

### auto-runner.ts 结构

```typescript
// 选项
interface AutoRunOptions {
  agent?: 'codex' | 'opencode';
  timeout?: number;       // ms, 默认 600000
  concurrency?: number;
  dryRun?: boolean;
  push?: boolean;
}

// 单任务结果
interface TaskExecResult {
  taskId: string;
  agent: string;
  result: 'ok' | 'error' | 'skip';
  durationMs: number;
  filesCreated: number;
  filesModified: number;
  error?: string;
  fallbackUsed?: boolean;
}

// 波次结果
interface WaveResult {
  wave: number;
  tasks: TaskExecResult[];
  commitHash?: string;
}

// 总结果
interface AutoRunResult {
  totalTasks: number;
  completed: number;
  failed: number;
  skipped: number;
  totalTimeMs: number;
  waves: WaveResult[];
}

// 主函数
export async function runAuto(projectDir: string, options: AutoRunOptions): Promise<AutoRunResult>

// 内部函数
async function executeTask(projectDir: string, taskId: string, target: string, timeout: number): Promise<TaskExecResult>
async function executeWave(projectDir: string, tasks: string[], options: AutoRunOptions): Promise<WaveResult>
function commitWave(projectDir: string, wave: WaveResult): string | null
```

### 关键实现细节

1. **spawn**: `child_process.execFileSync` 或 `execFile` + Promise 包装
   - bundle 写入 `os.tmpdir()/ham-bundle-{taskId}.txt`
   - 通过 `{ input: fs.readFileSync(bundlePath) }` 传 stdin
   - 超时用 `{ timeout }` 参数

2. **并行**: `Promise.allSettled(tasks.map(t => executeTask(...)))`

3. **git commit**: `execSync('git add file1 file2 && git commit -m "..."')`
   - 只 add task.files 中的文件
   - 如果 git add 没有变更（文件不存在），跳过 commit

4. **进度**: 直接 `process.stdout.write` 实时输出

5. **异步入口**: cmd-execute.ts 中 `execute auto` 返回 Promise
   - index.ts 的 main() 需要支持 async handler

## 四个层次的完整方案

### Layer 1: 执行自动化 (P0 — 本次实现)

即上述 `execute auto` 核心循环。

### Layer 2: 质量门禁 (P0 — 本次实现)

每波完成后、commit 前，做轻量验证：

```
验证项:
  1. 文件存在性: task.files 中的文件是否都创建/修改了
  2. 文件非空: 产出文件 > 0 bytes
  3. 语法检查: .ts/.tsx 文件尝试 `npx tsc --noEmit` (如果项目有 tsconfig)
  4. 导入检查: 扫描 import 的模块是否存在
```

验证失败的任务不 commit，标记为 failed，触发 fallback 或 skip。

不做的质量检查（留给人工审查）：
- 业务逻辑正确性
- 代码风格
- 测试覆盖率

### Layer 3: 限额调度 (P1 — 本次实现基础版)

```
场景: codex 连续失败 2 次（可能限额耗尽）
处理:
  1. 标记 codex 为 cooldown，记录时间
  2. 后续任务全部切到 opencode
  3. 30 min 后自动解除 cooldown，恢复正常路由
  4. 如果 opencode 也 cooldown → 暂停等待，每 5 min 重试一次
```

状态文件: `.ham-autocode/dispatch/agent-status.json`
```json
{
  "codex": { "status": "available", "lastFailure": null, "cooldownUntil": null },
  "opencode": { "status": "cooldown", "lastFailure": "2026-04-15T...", "cooldownUntil": "2026-04-15T..." }
}
```

### Layer 4: DAG 质量预检 (P1 — 本次实现基础版)

`execute auto` 启动前先做一轮 DAG 质量检查：

```
[auto] Pre-flight check...
[auto] task-006: spec.interface ✓, files declared ✓, deps resolved ✓
[auto] task-008: spec.interface ✓, files declared ✓, deps resolved ✓
[auto] task-011: spec.interface ✓, files declared ✓, deps resolved ✓
[auto] ⚠ task-013: spec.interface MISSING — bundle 质量可能不足
[auto] Pre-flight: 3 ready, 1 warning
[auto] Continue? (auto-yes in 5s)
```

检查项:
- task.spec.description 非空
- task.spec.interface 非空（对 codex/opencode 至关重要）
- task.files 已声明
- task.blockedBy 中的依赖都已 done
- 依赖任务的 spec.interface 非空（阅读清单需要）

缺失 spec 的任务标记 warning，仍然执行（agent 可能自己能搞定），但汇总报告中高亮。
